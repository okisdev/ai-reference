# Build the PR

Invoked from ship-pr step 2 to build the PR before the monitor loop. Resume at the sub-step the entry point selected (implement, branch and commit, or PR creation); skip this whole file when resuming an existing PR. Both create-side paths (plain git and GitButler) are below; detect which once via Current branch versus `gitbutler/workspace`, as in SKILL.md step 1.

1. **Implement and validate (only when entering from a task description).** Make the requested change; this is the one task-specific step ship-pr does directly. If it originates from an issue, optionally run `verify-issue` first to confirm the work is warranted. Validate with the repo-appropriate lint, build, and test commands auto-detected from the repo (package.json scripts, a Makefile, the CI workflow), preferring the narrowest checks that cover the change. Do not open a PR on a failing build.

2. **Branch and commit.** Plain-git path (Current branch is not gitbutler/workspace): first `git fetch origin` so the base is current (the plain-git analogue of `but pull`); if Current branch is the default base, invoke `make-branch` (it creates the branch off the freshened base, moves all uncommitted changes, honors `--base`, and stages but does not commit), otherwise stay on the existing feature branch; then `commit-changes` groups the staged work into conventional commits; then `git push -u origin HEAD`. Never inline the stash, checkout, pop, or commit-grouping. GitButler path (Current branch is exactly gitbutler/workspace): `but pull` to sync, `but branch new <branch-name>`, `but stage <file-or-hunk> <branch>` (if a stage id is ambiguous, run `but status -j` and use the longer id or the file path), `but commit <branch> --only -m "<message>"`, `but push <branch>`.

3. **Changeset (only if the repo uses one).** If the Changesets probe shows a `.changeset/` directory and a published package changed, follow the repo's convention: read `.changeset/config.json` and a recent `.changeset/*.md` for the package list and bump shape, then add one entry mapping each changed published package (package.json not `"private": true`, determined by the field rather than a hardcoded exemption list) to the bump level the repo conventionally uses (default the lowest, for example patch, when unsure), with a one-line summary of the user-facing change. Commit and push the changeset so it lands in the PR: `commit-changes` then push on the plain-git path, `but stage` plus `but commit` then `but push <branch>` on the GitButler path; never leave it uncommitted for make-pr to drop. If there is no `.changeset/` directory, skip the step entirely.

4. **Open the PR.** Plain-git path: delegate to `make-pr`, passing `--base` and `--draft` through; if a PR already exists it prints the URL and stops, which ship-pr treats as the signal to skip creation and continue. GitButler path: do not delegate to `make-pr` (it keys off `git branch --show-current` and pushes `HEAD`, which on gitbutler/workspace is the integrated workspace commit, not the virtual branch); instead run `gh pr create --head <virtual-branch>` against the branch `but push` created, adding `--draft` when requested, and capture the number with `gh pr view <virtual-branch> --json number`. Capture the PR number for the loop. Never reimplement title or body logic.
